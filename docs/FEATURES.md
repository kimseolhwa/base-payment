# 📋 Base Payment 단계별 기능 명세서

본 문서는 Base Payment 프로젝트의 도메인별 핵심 목적과 상세 구현 기능을 단계별(Phase)로 정의합니다.

## Phase 1: DB 조회 API 도메인 (`inquiry`)
**목적:** 저장된 결제 원장(DB)에서 결제 내역을 빠르고 안전하게 조회하는 코어 API 구축

* **결제 내역 단건 조회:** 가맹점 ID(`merchantId`)와 주문 번호(`orderId`)를 기반으로 결제 상세 내역 조회 수행.
* **불변 객체 기반 DTO 설계:** Java 21의 `record`를 도입하여 데이터 불변성 보장 및 보일러플레이트 코드 최소화.
* **파라미터 유효성 검증:** `@Valid`를 활용한 필수 파라미터 누락 및 형식 오류 방어 로직 구현.
* **데이터 검증 예외 처리:** 조회된 데이터가 없을 경우 `DataNotFoundException`을 발생시켜 글로벌 에러 핸들러를 통한 400 상태 코드 응답 처리.

---

## Phase 2: 노티 발송 API 도메인 (`noti`)
**목적:** 외부 시스템(가맹점 서버 등)으로 결제 결과를 통보하되, 외부 장애가 내부 시스템으로 전파되는 것을 차단

* **결제 결과 노티 발송:** 결제 완료 데이터를 외부 API로 전송 및 통신 결과 DB 이력 적재.
* **트랜잭션 범위 최적화:** 외부 API 연동(I/O) 구간을 DB 트랜잭션과 물리적으로 분리하여, 통신 지연 시 DB 커넥션 고갈(Deadlock) 현상 원천 차단.
* **서킷 브레이커(Circuit Breaker) 방어 체계:** `Resilience4j`를 적용하여 외부 서버 장애 또는 타임아웃 발생 시 즉각 통신을 차단(Fail-Fast)하고 Fallback 응답 반환.

---

## Phase 3: 결제 인증 UI 연동 도메인 (`cert`)
**목적:** 사용자(End-User)가 직접 마주하는 결제 인증 화면을 렌더링하고, 승인 전까지의 결제 세션을 안전하게 유지

* **결제 인증 UI 렌더링:** `Thymeleaf` 템플릿 엔진을 활용하여 동적인 사용자 결제 확인 화면 제공.
* **임시 결제 세션 관리:** 인증 페이지 진입 시 생성된 결제 파라미터(금액, 가맹점 정보 등)를 `Redis`에 임시 저장 (유효기간 TTL 5분 설정).
* **최종 결제 승인 처리:** 사용자가 화면에서 '승인' 요청 시, Redis에 저장된 세션 데이터를 검증하여 최종 결제 처리.
* **중복 결제 방지 (멱등성):** 승인 요청 시 Redis 분산 락(Distributed Lock)을 획득하여 동일한 사용자의 동시 다발적인 클릭(따닥 요청) 완벽 방어.

## 📌 API 개발 가이드 및 제약사항

프로젝트의 통일성 있는 API 명세 관리를 위해 아래 가이드를 준수합니다.

* **OpenAPI(Swagger) 적용 필수:** 앞으로 추가되는 모든 API 컨트롤러와 DTO에는 반드시 `@Tag`, `@Operation`, `@Schema` 등의 `springdoc-openapi` 애노테이션을 부착하여 스펙을 명시해야 합니다.
* **환경별 Swagger 노출 정책:** API 스펙 노출에 따른 보안 이슈 방지를 위해 **운영(`real` 또는 `prod`) 환경에서는 Swagger UI가 비활성화**됩니다. 로컬(`local`), 개발(`dev`), 테스트(`test`) 환경에서만 접근을 허용합니다.
